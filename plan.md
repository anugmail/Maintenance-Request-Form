# แผนรีวิว/เดโม prototype "แจ้งซ่อม–กบค." — 6 หมุดหมาย (Review Milestones)

> เอกสารรายละเอียดการทำของแต่ละหมุด คู่กับหน้ารีวิว [05-review-milestones.html](05-review-milestones.html) (การ์ดบนฮับ `index.html`)
> — หน้า 05 ใช้เดโมกับเจ้าของงาน · ไฟล์นี้ใช้ฝั่งทีมทำ (รายละเอียดจอ/validation/ข้อมูล/จุด verify)

**Goal:** หั่น flow แจ้งซ่อม–กบค. (ใน `mock/Maintenance-Request-Form.html`) ออกเป็น **6 หมุดรีวิว/เดโม** เดินรีวิวกับเจ้าของงานทีละช่วง เก็บคำตอบของคำถามเปิดกลับมาปรับ mock ทีละรอบ

**Architecture:** Static HTML ไฟล์เดียว + vanilla JS (ไม่มี build) · 7 view สลับบทบาทผ่าน `switchView()` (`form / my / fuel / kry / kbk / sup / stock`) · เรื่องแจ้งซ่อมเป็น object ใน `JOBS[]` เดินสถานะด้วย `phase` (`self → kry → krd/pending → work → slots → appt → approve → closed` + เส้นอู่ `returned → gclosed`) · deep-link ด้วย hash (`#kry`) + `?seed=review` เติมงานตัวอย่างเฉพาะรอบเดโม (ไม่กระทบ default)

**ขอบเขต:** เฉพาะ flow แจ้งซ่อม–กบค. (ไม่รวม PWA บันทึกการใช้รถ / บำรุงรักษาตามวาระ)

# หมุด M1 — สร้างใบแจ้งซ่อม (ผู้แจ้ง) — รายละเอียดการทำ

## จุดเข้าเดโม

- M1 ไม่ต้องใช้ `?seed=review` — เริ่มจากฟอร์มว่างแล้วสร้างเรื่องจริงได้เลย · ปุ่มรอง "ติดตามสถานะเรื่องของฉัน" → `#my`

## โครงฟอร์ม: wizard 4 ขั้น

Stepper อยู่ใน `#wsteps` เดินตาม `WORDER = ['vehicle','symptom','parts','decision']` — **สลับลำดับ/เปิดปิดขั้นได้จาก Admin** (`cfg.steps.wizard`) · ปุ่มท้ายฟอร์ม `ย้อนกลับ/ถัดไป` (`nav(±1)`) ขั้นสุดท้ายเปลี่ยนเป็น "ส่งเรื่องแจ้งซ่อม"

### ขั้น 1 — เลือกรถ (`s1`)

| เรื่อง | รายละเอียด |
|---|---|
| ค้นหา | ทะเบียน/รุ่น จากรายการ "รถของฉัน/รถในทีม" (`VEHICLES` จาก master data — Admin แก้ได้) · รถไม่อยู่ในระบบ → ติดต่อผู้ดูแลข้อมูลรถ (กรย./กบค.) |
| การ์ดรถ | ใช้ shared component `UIC.vehicleCard` (variant สลับได้จาก Admin) |
| ข้อมูลอัตโนมัติ | เลือกแล้วระบบดึง ยี่ห้อ/รุ่น · สังกัด · อุปกรณ์เสริม ให้เอง (read-only) |
| จุดที่มีปัญหา | radio **ตัวรถ / อุปกรณ์เสริม** (ตัวเลือกอุปกรณ์เสริมโผล่เฉพาะรถที่มี เช่น กระเช้า/เครน) — เปลี่ยนรถ/เปลี่ยนจุด → ล้างอาการ+การจองที่เลือกไว้ |
| Validation | ต้องเลือกรถ + จุดที่มีปัญหา ก่อนไปต่อ |

### ขั้น 2 — อาการเสีย (`s2`)

| เรื่อง | รายละเอียด |
|---|---|
| รายการอาการ | chip มาตรฐาน 11 อาการ ใน 5 หมวด (`CATS`): ไฮดรอลิก · บูม/กระเช้า · ไฟฟ้า · เครื่องยนต์ · อื่นๆ — **กรองตามจุดที่เลือก** (หมวดมี `for: vehicle/attach/both`) เลือกได้หลายอาการ |
| อื่นๆ | เลือก `OTH` → เปิด textarea บังคับพิมพ์อธิบาย |
| ความรุนแรง | radio 4 ระดับ: ต่ำ / ปานกลาง / สูง / หยุดใช้งาน (`SEVS`) |
| รูปถ่ายอาการ | แนบไฟล์จริง (multiple) หรือปุ่ม "รูปตัวอย่าง" สำหรับเดโม · ลบได้ · ปิดทั้งช่องได้จาก Admin (`TG.photoUpload`) |
| Validation | ≥1 อาการ · เลือก OTH ต้องกรอกข้อความ · ต้องเลือกความรุนแรง |

### ขั้น 3 — อะไหล่ที่ระบบแนะนำ (`s3`)

| เรื่อง | รายละเอียด |
|---|---|
| Suggest | map อาการ → อะไหล่ (`PARTS.sym`) แสดง รหัส/ชื่อ/จำนวนที่ต้องใช้ต่อครั้ง/คลังที่เก็บ/เวลาอัปเดตสต็อก |
| Badge สถานะคลัง | `พร้อมเบิก · เหลือ N` / `ใกล้หมด (≤3)` / `หมด (+ETA ถ้ามี)` |
| มีของ | ปุ่ม **จองอะไหล่** — ปรับจำนวน ± ได้ ไม่เกินสต็อก · ยกเลิกได้ |
| ของหมด | ปุ่ม **สั่งของ** — เห็น ETA ของเข้า · รายการไปโผล่หน้า "คลังอะไหล่" (`#stock`) ฝั่ง กบค. |
| Auto-cleanup | ถอดอาการออก → ยกเลิกจอง/สั่งของของอาการนั้นให้อัตโนมัติ (+toast แจ้ง) |
| อาการอื่นๆ | ไม่มีอะไหล่แนะนำ — ให้ช่าง กบค. ประเมินหลังรับเรื่อง |
| หมายเหตุ | mock ยังไม่ตัดสต็อกจริง (แสดงเป็นการจอง) |

### ขั้น 4 — ตัดสินใจ + สรุป (`s4`)

| เรื่อง | รายละเอียด |
|---|---|
| ทางเลือก | 2 tile: **ซ่อมเอง** (กรอกแผน/บันทึกโดยย่อ) หรือ **ส่งเรื่องให้ กรย.** (กรอกเหตุผลที่ซ่อมเองไม่ได้) |
| กล่อง routing | อธิบายเส้นทาง: กรย. (กองระบบไฟฟ้าและงานโยธา 12 เขต) จะคัดแยก **ซ่อมอู่ / กรย. ซ่อมเอง / ส่งต่อ กบค.** — อาการ รูป อะไหล่ที่จอง/สั่ง แนบไปอัตโนมัติ |
| ตารางสรุป | รถ · จุดที่มีปัญหา · อาการ · ความรุนแรง · รูปแนบ · อะไหล่จอง/สั่ง · ทางที่เลือก |
| Validation | ต้องเลือกทางใดทางหนึ่งก่อนส่ง |

## ผลลัพธ์เมื่อส่งเรื่อง (`submit()`)

- ออก**เลขที่ใบแจ้งซ่อม** `MTD-690717-0NN` (running `SEQ`) แสดงหน้าจอสำเร็จ + ปุ่มไป "เรื่องของฉัน" / แจ้งเรื่องใหม่
- สร้าง job เข้า `JOBS[]` เก็บ: เลขที่ · รถ+อุปกรณ์เสริม · สังกัด · จุดที่มีปัญหา · อาการ (รวมข้อความอื่นๆ) · ความรุนแรง · จำนวนรูป · คำอธิบาย · อะไหล่จอง/สั่ง (พร้อม ETA) · เหตุผล/แผน · `history[]` timeline
- **ส่ง กรย.** → `phase:'kry'` เข้าคิว "งานคัดแยก กรย." ทันที (badge เมนูเด้ง) = จุดส่งไม้ต่อเข้า **M2**
- **ซ่อมเอง** → `phase:'self'` บันทึกเข้าประวัติรถ พร้อม snapshot อะไหล่
- ผู้แจ้งติดตามที่ "เรื่องแจ้งซ่อมของฉัน" (`#my`) — เปิดดูรายละเอียด+timeline แบบ accordion

## สคริปต์เดโมแนะนำ (~2 นาที)

1. เปิด M1 จากหน้า 05 → ค้นรถ "83-xxxx" เลือกรถกระเช้า → ชี้ข้อมูลอัตโนมัติ → เลือกจุด "กระเช้า"

# หมุด M2–M6 — โครงย่อ (จะขยายรายละเอียดเมื่อถึงรอบรีวิวของหมุดนั้น)

## M2 — คัดแยก + รับเข้า กบค. (`?seed=review#kry`)
กรย. เห็นคิวงานคัดแยก (seed 1 เรื่อง) → เลือกช่องทาง **ซ่อมอู่ (+แนะนำอู่ใกล้เคียง) / กรย. ซ่อมเอง / ส่งต่อ กบค.** → เรื่องเข้าคิว กบค. → กบค. **รับไว้ซ่อม / ส่งกลับ** (+เหตุผล +แนะนำอู่)
คำถามเปิด: เส้นทางจริงผ่าน กรย. หรือ "แผนกยานพาหนะของกอง" (flow#10) · Overhaul / จำหน่าย / จ้างเอกชน (flow#2,3,4)

## M3 — ประเมิน + จัดการอะไหล่ (`?seed=review#kbk`)
ประเมินอาการ+ราคา → เดินสถานะ รอสั่งอะไหล่ → อะไหล่ถึงแล้ว (รายการ+ETA) → หน้า "คลังอะไหล่": จอง / สั่งของ / คืน
คำถามเปิด: เบิกจากคลังไหน (flow#1) · เงื่อนไขดัดแปลงอะไหล่ (flow#9)

## M4 — นัดหมาย + รับรถ + ตรวจสภาพขาเข้า (`?seed=review#kbk`)
กบค. เปิดตารางช่วงว่าง → ผู้แจ้งเลือกยืนยัน 1 ช่วง → **ตรวจสภาพขาเข้า** (โซนรอบคัน/น้ำมัน/ไมล์/อาการเดิม) บังคับก่อนเริ่มซ่อม
คำถามเปิด: ซ่อม สนญ. vs ออกหน้างาน · ใบตรวจสภาพตอนออกหน้างาน (flow#5)

## M5 — ดำเนินการซ่อม + บันทึกผล (`?seed=review#kbk`)
เดินสถานะซ่อม (5 ขั้น `KBK_STEPS`) → เจออาการเพิ่ม → **ผู้แจ้งกดรับทราบ** → ตรวจขาออกเทียบขาเข้า → คืนอะไหล่ → บันทึกประวัติ
คำถามเปิด: "หน่วยงานกดรับทราบ" = ใคร ไม่กดบล็อกไหม (flow#8)

## M6 — ต้นทุน + ปิดงาน + อนุมัติ (`?seed=review#sup`)
คำนวณต้นทุน → รออนุมัติ → ผู้อนุมัติ (**ผู้บังคับบัญชาของ กบค.** — ยืนยันแล้ว) อนุมัติ/ตีกลับ + timeline → ปิดงาน + แจ้งทุกฝ่าย
**เส้นอู่/จ้างเอกชน (ซ่อมนอกระบบ):** ช่วงเสนอราคา→ซ่อม อยู่นอกระบบ — กลับมา**กรอกผลซ่อม + วันที่ซ่อมเสร็จ + แนบ ใบแจ้งหนี้/ใบเสร็จ + เอกสารผู้รับจ้าง ครบก่อนจึงปิดงานได้** (gate บังคับ — ทำแล้วใน mock)
คำถามเปิด: สูตร "SUM ต้นทุน" (flow#6) · เกณฑ์ e-GP (ยังไม่ชัด — ข้ามไว้ก่อน)
